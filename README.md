# EvasionRL

**Live demo:** https://arashvsh-evasionrl-app-najahp.streamlit.app/

**EvasionRL** is an interactive Streamlit demo that shows how a trained reinforcement-learning robot can be disrupted by an **observation-space evasion attack**.

The robot is controlled by a trained **Deep Q-Network (DQN)**. In the clean setting, the DQN receives the true observation from the environment. In the attacked setting, the environment still behaves normally, but the observation is slightly modified before it reaches the DQN.

The purpose of the app is to make this security idea visible:

> A robot can fail even when its model, reward function, and environment physics are unchanged. It may be enough to slightly change what the robot sees.

---

## 1. Project overview

EvasionRL compares two versions of the same reinforcement-learning task side by side:

- **Clean robot:** receives the true observation from the environment.
- **Attacked robot:** receives a perturbed observation generated by an evasion attack.

Both robots use the same trained DQN policy. The only difference is the observation passed into the model.

The app lets users adjust the attack strength, run the simulation, watch both robots move, and then read a short final summary explaining what happened.

---

## 2. Tutorial: the task

The project uses the classic **MountainCar-v0** reinforcement-learning environment.

In this task, a small car is stuck in a valley between two hills. The car is not powerful enough to drive directly up the right hill. To reach the flag, it must learn to move back and forth, build momentum, and then climb the hill.

### Observation

At every timestep, the environment gives the agent a two-number observation:

```text
[position, velocity]
```

- **position** tells where the car is on the track.
- **velocity** tells how fast and in which direction the car is moving.

Example:

```text
[-0.445, 0.020]
```

This means the car is around position `-0.445` and moving slightly to the right.

### Actions

The DQN chooses one of three actions:

```text
0 = push left
1 = no push
2 = push right
```

### Goal

The goal is to reach the flag on the right side of the hill.

In MountainCar, the target is approximately:

```text
position >= 0.5
```

### Reward

The environment gives a reward of `-1` at each step until the task ends. So a better agent reaches the flag in fewer steps and gets a less negative total return.

For example:

```text
Reached flag in 110 steps → return ≈ -110
Failed for 300 steps       → return ≈ -300
```

---

## 3. Tutorial: the DQN agent

The agent is a **Deep Q-Network**.

A DQN receives the observation and predicts a Q-value for each possible action.

For MountainCar, the input is:

```text
[position, velocity]
```

The output is:

```text
Q(push left), Q(no push), Q(push right)
```

The agent chooses the action with the highest Q-value.

Example:

```text
Q(push left)  = 1.2
Q(no push)    = 0.8
Q(push right) = 2.4
```

The DQN chooses:

```text
push right
```

The key point is that the DQN acts only based on the observation it receives. If the observation is manipulated, the DQN may choose a bad action even though the physical environment has not changed.

---

## 4. Tutorial: what the agent sees

The clean robot sees the true observation from the environment:

```text
environment → true observation → DQN → action
```

The attacked robot has a different pipeline:

```text
environment → true observation → attack → modified observation → DQN → action
```

The environment still keeps the real physical state of the car. The attack only changes the numbers shown to the DQN.

Example:

```text
True observation:      [-0.445,  0.000]
Perturbation:          [-0.050, +0.050]
DQN receives:          [-0.495,  0.050]
```

So the attacked robot is not physically moved by the attack. Instead, the DQN is given a misleading observation and may choose the wrong action.

---

## 5. Tutorial: evasion attacks

An **evasion attack** happens at test time or deployment time.

It does not retrain the model. It does not poison the training data. It does not modify the reward function. It only modifies the input observation before the trained model uses it.

In this project, the threat model is:

- The attacker can slightly modify the robot's observation.
- The attacker cannot change the trained DQN weights.
- The attacker cannot change the reward.
- The attacker cannot change the environment physics.
- The attacker cannot directly force an action.

This is an observation-space evasion attack.

---

## 6. Attacks included in the app

### No attack

The attacked robot receives the true observation. This is useful as a baseline.

### Random noise

The app adds small random noise to the observation.

This checks whether the robot is sensitive to ordinary noise.

### FGSM

FGSM stands for **Fast Gradient Sign Method**.

It is a one-step gradient-based attack. It uses the DQN's gradient to find a harmful direction in the observation space, then shifts the observation slightly in that direction.

Simple idea:

```text
Find a small change to [position, velocity] that makes the DQN prefer a worse action.
```

FGSM is fast and easy to visualize.

### PGD

PGD stands for **Projected Gradient Descent**.

It is a stronger multi-step version of FGSM. Instead of making one change, it repeatedly takes small harmful steps and keeps the final perturbation inside the allowed attack budget.

Simple idea:

```text
small harmful step → clip to allowed range → repeat
```

PGD is usually stronger than FGSM.

---

## 7. Attack strength: epsilon

The sidebar control **epsilon** controls the maximum allowed observation perturbation.

Small epsilon means the attack can only make tiny changes.

Large epsilon means the attack can make stronger changes.

Example with epsilon `0.05`:

```text
True observation:      [-0.445,  0.000]
DQN receives:          [-0.495,  0.050]
```

The perturbation is bounded so the attacker cannot arbitrarily replace the observation.

## 8. What the app does exactly

The Streamlit app runs a side-by-side comparison.

### During the simulation

The app shows two robot views:

- **Clean robot** on the left.
- **Attacked robot** on the right.

The app does not show long timestep logs during the run. This keeps the animation smooth and makes the demo easier to follow.

The clean robot receives the true observation. The attacked robot receives the perturbed observation.

### Max demo steps

The app keeps running until the selected **Max demo steps** limit is reached or both robots finish.

If the clean robot reaches the flag early, its view freezes as finished. The attacked robot keeps running until it also finishes or the selected step limit is reached.

This is important because the clean robot may succeed quickly, while the attacked robot may need more time to show failure.

The app also overrides MountainCar's default 200-step limit, so the sidebar value is the actual experiment limit.

### After the simulation

After the run finishes, the app shows a compact final summary.

The summary includes:

- whether the clean robot reached the flag
- whether the attacked robot reached the flag
- each robot's final position
- each robot's final velocity
- each robot's total return
- when each robot finished
- the attack type
- epsilon
- the last true observation
- the last observation seen by the DQN
- the perturbation added to the observation
- whether the attack changed the attacked robot's decision
- whether the clean and attacked robots chose different actions while both were active

## 9. How to interpret the result

A strong attack demonstration usually looks like this:

```text
Clean robot:    reaches the flag
Attacked robot: fails to reach the flag
```

This means the trained DQN can solve the task normally, but the observation attack disrupts its behavior.

A weaker or unclear attack demonstration may look like this:

```text
Clean robot:    reaches the flag
Attacked robot: also reaches the flag
```

In that case, the attack did not clearly prevent success in that episode. Try increasing epsilon, using PGD, changing the seed, or increasing the max demo steps.

If both robots fail, the clean policy may not be strong enough. Retrain the DQN for more timesteps.

## 10. Recommended demo settings

Recommended starting settings:

```text
Attack type:      PGD
Epsilon:          0.05 to 0.15
PGD steps:        20
Objective:        flip_margin
Max demo steps:   300 to 1000
Frame delay:      0.02 to 0.05
```

```bash
python -m venv .venv
```

```bash
source .venv/bin/activate
```

```bash
pip install -r requirements.txt
```

## 12. Train the DQN agent

Train the model:

```bash
python train_dqn.py --timesteps 500000
```

If the clean robot is still weak, train longer:

```bash
python train_dqn.py --timesteps 1000000
```

The training script saves the model to:

```text
models/evasionrl_dqn_mountaincar.zip
```

It also saves a training summary to:

```text
models/evasionrl_dqn_mountaincar.training_summary.json
```

The app reads this summary and warns users if the clean model appears weak.

---

## 15. Key takeaway

EvasionRL shows that a trained reinforcement-learning robot can fail when its observations are slightly modified at deployment time.

The robot is not hacked by changing its model or reward. It is attacked by changing what it sees.

That is the central lesson of the project.
